### Go 1.26.5 comparison

The same `d20cf95` source was compiled with exact Go 1.25.0 and 1.26.5 toolchains, `CGO_ENABLED=0`,
release linker flags, and isolated build caches. Single-threaded benchmarks were pinned to one CPU;
startup used 400 runs per toolchain in reversed order, and the six-core in-memory Processor core
used 100 runs per toolchain in reversed order. Positive deltas mean faster execution; there is no
aggregate percentage across unrelated workloads.

| Identical-source metric | Go 1.25.0 | Go 1.26.5 | Speed delta |
|---|---:|---:|---:|
| No-prompt cold start | 41.419 ms | 42.105 ms | **-1.66%** |
| Minimal `AgentSession` creation | 16.408 ms | 17.188 ms | **-4.75%** |
| In-memory Processor core, 1,000 turns / 100 conversations | 205.434 ms | 199.702 ms | **+2.79%** |
| F12 full-screen frame replay | 292.100 µs | 283.970 µs | **+2.78%** |
| Latest compaction, 20,000 entries | 204.336 µs | 205.345 µs | **-0.49%** (noise-sized) |
| Stripped Linux/amd64 binary | 50,569,400 B | 51,024,034 B | **+0.90% size** |

F12 also drops from 663 to 662 allocations per frame; session creation remains about 38,726
allocations and compaction remains allocation-free. The migration is for the current supported Go
release and its fixes, while the measured performance result is mixed: the concurrent gateway and
renderer improve, but startup and session construction do not.

The 2026-07-21 exact-tree refresh consolidates the duplicated locale resolver and branch-summary
construction while wiring upstream's skill renderer. The final dead-code pass removed three unused
entry points and their state: a superseded CLI input wrapper, an unearned skill sorter, and unused
internal semver string storage. `staticcheck U1000` is clean; deadcode's remaining 25 reports are
exported upstream/SDK entry points, so deleting them would remove supported surface.

The durable benchmark separates the in-memory Processor concurrency number from production I/O.
In a profile containing the 1,000-turn seed plus one durable turn, `LocalProvider.Acquire` accounts
for about 78.2 MB and the ledger scan for 4.9 MB; the final 10-turn series reports 109.7 MB/op.
Removing the per-append ID-index copy drops the identical whole-profile allocation space from
724.26 MB to 142.22 MB, an **80.4% reduction**; the timed long turn itself drops about 1.1%, so no
unsupported claim is made that JSONL hydration disappeared.

The fresh release measurement initially failed at 55.7 ms on an isolated CPU with 25
auto-discovered skills. A syscall trace found 4,118 file-stat calls: canonical-path deduplication
re-resolved every earlier path for each new skill, then the default loader resolved the same
package metadata again. A local seen set and reuse of the already-resolved metadata preserve
first-path-wins and reload freshness while reducing minimal session creation from 48–50 ms to
31–33 ms, 6.48 MB to 5.70 MB, and about 48,940 to 38,730 allocations. The original cold-start
harness now passes above; the controlled before/after is 55.7 → 40.9 ms.
