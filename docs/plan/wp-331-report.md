## Behavior

`pi --mode rpc` now serves upstream's bidirectional JSONL protocol over stdin/stdout. Input is split
only on LF, CRLF is accepted by removing the trailing CR, a final unterminated record is processed,
and every response or session event is serialized as one atomic line. Prompt success is emitted
after preflight and before streaming begins, while a preflight failure emits exactly one error
response. Closing stdin cancels in-flight commands before shutdown waits for them, so a running bash
command cannot hold the process open. Prompt execution is reserved before its success response is
observable, so an immediate `streamingBehavior` prompt is queued instead of racing a second run.

The command surface covers prompt, steer, follow-up, abort, model and thinking controls, queue
modes, compaction and retry controls, bash execution, session statistics and export, session
replacement/fork/clone, entries/tree/message queries, naming, command discovery, and unknown-command
errors. New, switched, and forked sessions replace the live runtime and rebind event forwarding.
Model cycling retains upstream's authenticated scoped-model behavior, including explicit per-model
thinking levels and the out-of-scope starting-index quirk.

The extension UI transport implements dialog request/response correlation plus the fire-and-forget
notify, status, widget, title, and editor-text variants. Optional empty strings and empty widget
lists remain distinguishable from absent fields, and each variant uses upstream's field order.
Phase 5 binds JS extensions to this transport; WP-331 owns the protocol surface itself.

Settings mutations used by RPC are lock-protected global-file writes. They re-read and merge the
current file, preserve unrelated member order, apply pending upstream migrations, retain literal
HTML characters, omit a trailing LF, and refuse to overwrite malformed settings.

## Conformance evidence

F7 is generated by the real pinned TypeScript RPC mode over the faux provider with fixed time,
UUIDs, session IDs, entry IDs, working directory, and response chunks. Its manifest declares no
canonicalization. The Go runner replays all 40 request steps and 54 output records through both
`RunRPCMode` and an actual `cmd/pi --mode rpc` executable, comparing every line byte-for-byte. The
binary is built with a conformance-only dependency seam that supplies the fixed faux provider,
clock, IDs, and cancelling session host used by the upstream extractor; CLI parsing, strict framing,
command dispatch, event forwarding, serialization, and shutdown are the production paths. Coverage
includes V8 parse-error text, the full prompt lifecycle, populated messages, entries, nested tree,
cursor queries, fork candidates, persisted bash entries, and leaf `children: []` shape.

An additional untagged production-binary test starts `pi --mode rpc` with a configured model and a
local OpenAI-compatible streaming server. It exercises stdin/stdout framing, state and empty-array
shapes, a streamed prompt through `message_end` and `agent_settled`, and clean EOF without the
conformance seam.

`make upstream-rpc-tests` installs pinned `vitest@4.1.9` as extraction-only tooling, builds the
pure-Go CLI, and places a temporary `dist/cli.js` adapter in the pinned upstream checkout.
Upstream's actual `RpcClient` first connects to the Go process and verifies state, then the six
upstream files run unmodified. The 17 `rpc.test.ts` process cases drive the Go binary; the other ten
cases validate TypeScript client, framing, clone, and process-exit behavior and therefore are not
independent Go parity evidence. A local Anthropic-compatible SSE server and per-process temporary
model configuration replace the external credential boundary. The complete upstream invocation
